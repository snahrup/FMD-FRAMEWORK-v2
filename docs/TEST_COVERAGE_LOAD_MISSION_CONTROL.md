## Test Data & Mocking Strategy

### Database Mocking (API Tests)

The API tests use Python's `unittest.mock` to mock database queries:

```python
@pytest.fixture
def mock_db():
    with patch('dashboard.app.api.routes.load_mission_control.db') as mock:
        yield mock
```

Tests can then define mock return values:

```python
mock_db.query.return_value = [
    {"source": "MES", "entity_count": 45},
    {"source": "M3ERP", "entity_count": 596},
]
```

This allows testing:
- ✅ Normal operation with sample data
- ✅ Empty result sets
- ✅ Null/missing values
- ✅ Database errors (Exception scenarios)
- ✅ Query parameter passing

### API Mocking (UI Tests)

Playwright tests use route interception to mock API responses:

```typescript
await page.route('**/api/lmc/**', (route) => {
  route.fulfill({
    status: 200,
    contentType: 'application/json',
    body: JSON.stringify({ /* response */ }),
  });
});
```

This allows testing:
- ✅ Success responses
- ✅ Error responses (500, 404, etc.)
- ✅ Network timeouts
- ✅ Malformed JSON responses
- ✅ Slow responses (loading states)

## Known Limitations & Future Enhancements

### Current Scope

The test coverage focuses on:
- ✅ Happy path and common error scenarios
- ✅ Data structure and contract validation
- ✅ User interaction workflows
- ✅ Basic performance metrics

### Out of Scope

The following require additional setup or are covered elsewhere:

- **Real Fabric Workspace Testing** — Tests use mocked data; production Fabric testing requires actual workspace access
- **Database Integration** — API tests use mocks; full DB integration tests in separate suite
- **Authentication** — Tests assume API is accessible without auth; OAuth/security testing separate
- **Load Testing** — Tests are functional; load/stress testing in separate suite
- **Visual Regression** — Requires screenshot baselines and MRI visual testing

### Recommended Future Tests

1. **Percy/Visual Regression Testing** — Capture visual state across viewports
2. **Performance Profiling** — React DevTools Profiler for render optimization
3. **Accessibility Audit** — axe-core for comprehensive a11y checks

## Code Quality & Standards

### Test Organization

Tests follow these standards:

✅ **Clear naming:** Test names describe what is being tested
✅ **Single responsibility:** Each test validates one thing
✅ **Setup/Teardown:** Fixtures handle setup and cleanup
✅ **Assertions:** Clear, meaningful assertions with context
✅ **Documentation:** Comments explain complex scenarios
✅ **No test interdependencies:** Tests run independently

### Assertions & Expectations

Tests use these assertion libraries:

- **Playwright:** `expect(locator).toBeVisible()`, etc.

## Maintenance & Updates

### When to Update Tests

Update tests when:

1. **Component API Changes** — New props, removed features
2. **Data Schema Changes** — New API fields, removed fields
3. **Workflow Changes** — New user interactions, changed behavior
4. **Error Handling Changes** — Different error messages, recovery paths

### When NOT to Update Tests

Keep tests as-is when:

- ✅ Implementation details change (CSS classes, internal state)
- ✅ Performance optimizations (same behavior, faster)
- ✅ Code refactoring (same functionality)
- ✅ Internal logging changes
